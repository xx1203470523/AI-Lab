
# Anti Patterns

> 通用 MD 写作原则遵守 [md-writing.md](md-writing.md)。

## Purpose

防止 AI 工程体系退化：规则污染、skill 膨胀、context 失控、结构混乱。

---

## 1. CLAUDE.md

| 禁止 | 正确 |
|------|------|
| 写完整架构 | 只放入口+约束+路由 |
| 写 workflow 全流程 | 指向 workflow 目录 |
| 写 skills/rules 内容 | 指向对应文件 |
| 写 Prompt 模板 | 不在 CLAUDE.md 出现 |

## 2. Context

| 禁止 | 正确 |
|------|------|
| 写规则（must/should） | 只写事实 |
| 写 skill（how to do） | 只写是什么 |
| 写实现细节、优化建议 | 只写架构结构+技术栈+模块 |
| 写 DDD 讲解、Prompt | 不在 context 出现 |

## 3. Rules

| 禁止 | 正确 |
|------|------|
| 写教程、解释性段落 | 只写可判断的约束 |
| 写设计理念、案例分析 | 短、明确、可执行 |
| 混入 skill/context | 一个 rule = 一个约束点 |

## 4. Skills

| 禁止 | 正确 |
|------|------|
| 写项目背景、架构说明 | 直接写 Trigger/Checklist/Fix |
| 写 Prompt、"你应该如何思考" | 流程化步骤 |
| 一个 skill 做多件事 | 一个 skill = 一个问题类型 |

## 5. 文件膨胀

| 禁止 | 正确 |
|------|------|
| 单个 md > 150 行（context 除外） | 一个文件 = 一个意图 |
| 一个文件多个主题 | 超限即拆分 |
| rule 写成百科全书 | 按职责拆 |

## 6. 跨层污染

禁止以下混合：

- skill 里写 rule
- rule 里写 skill
- context 里写 rule
- workflow 里写 skill

## 7. Prompt 化

任何文件中禁止出现：

- "你是一个专业AI"
- "请一步一步思考"
- "请扮演..."
- "作为一个资深工程师"

> 这些属于 Prompt，不属于工程结构。

## 8. 全量加载

| 禁止 | 正确 |
|------|------|
| 启动时读取整个 ai-lab | 按 context-map 路由 |
| 无路由直接扫描 | 按需加载 skill/rule |

## 9. Skill 膨胀

一个 skill 不能包含：SQL + 并发 + 架构 混在一起。超过一个领域 → 拆分。

## 10. Rule 膨胀

一个 rule 文件不能包含所有规范。不能写成架构文档或设计指南。

---

## 核心原则

| 层 | 职责 | 判断标准 |
|---|------|----------|
| context | 是什么 | 事实陈述 |
| rules | 不能做什么 | 可判断、可执行 |
| skills | 怎么做 | 流程化步骤 |
| standards | 怎么写这些文件 | 结构约束 |

**最重要：任何文件只允许服务一个目的。**

如果发现文件越来越长、内容越来越杂、skill 越来越像论文 → 体系正在退化，必须拆分。
