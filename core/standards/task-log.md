# 任务日志学习笔记

## Purpose

本文记录 AI 协作中的任务日志模式，用于追溯一次任务做了什么、为什么这样做、改了哪些文件。

任务日志不是业务知识库，但它能补足 git diff 和 commit message 无法表达的上下文。

---

## 1. 为什么需要任务日志

git 能告诉我们代码改了什么，但不一定说明：

- 当时用户为什么提这个需求。
- AI 参考了哪些约束。
- 采用了什么判断。
- 哪些验证没有执行。
- 后续维护者应该注意什么。

任务日志用于记录这些“执行上下文”。

---

## 2. 推荐结构

```text
logs/{git-user}/{date}/{branch}/{scope}/{task-name}.md
logs/INDEX.md
```

字段含义：

| 字段 | 说明 |
|------|------|
| `git-user` | 当前 git 用户 |
| `date` | 日期 |
| `branch` | 分支名，斜杠替换为短横线 |
| `scope` | 全局 / 后端 / Web / PDA |
| `task-name` | 简短中文任务名 |

---

## 3. 单个日志内容

建议包含：

```markdown
# 任务名

- 日期
- 分支
- 目录范围
- 对话轮次

## 做了什么

## 改动文件

## 执行逻辑
```

重点是写清：

- 需求背景。
- 关键决策。
- 文件影响范围。
- 验证情况。

---

## 4. INDEX.md 的价值

`logs/INDEX.md` 是任务日志索引。

它的作用：

- 新任务开始前快速查历史。
- 写 commit message 时复用任务语义。
- 避免重复踩坑。
- 让 AI 在长周期项目中保留上下文。

索引摘要应包含可搜索关键词，不要只写“修复 bug”。

---

## 5. Stop Hook 模式

工程化项目可以用 Stop hook 检查：

```text
如果本轮调用过 Edit/Write
且最近没有任务日志
则提醒补写日志
```

优点：

- 防止 AI 忘记记录。
- 形成稳定工作闭环。

注意：

- Stop hook 要避免死循环。
- 低价值纯文档探索可以不强制。
- 用户明确不需要日志时，应谨慎处理项目规则和用户意图的冲突。

---

## 6. 什么时候需要日志

建议写：

- 修改代码。
- 修改配置。
- 调整规则。
- 解决冲突。
- 做出重要技术决策。

可以不写或简写：

- 纯问答。
- 临时只读查询。
- 未产生实际改动的探索。

---

## 7. 与 ai-lab 的关系

- logs 记录“本次任务发生了什么”。
- ai-lab 记录“未来还会复用的知识”。

如果某条日志里的经验反复出现，再提炼到 ai-lab。
