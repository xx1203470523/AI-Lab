# 规则生命周期

## Purpose

本文用于判断一条经验、规则或流程应该放在哪里：README、rules、quick-ref、skills、hooks、agents，还是 logs。

目标是避免 ai-lab 越写越乱，也避免过早工程化。

---

## 1. 规则从哪里来

规则通常来自：

- 用户反复提醒。
- 生产事故。
- 代码审查问题。
- 多次任务中的重复模式。
- 业务链路中代码看不出来的事实。
- 团队工程化配置中验证有效的约束。

---

## 2. 放置位置判断

| 经验类型 | 推荐位置 |
|----------|----------|
| 一次性任务过程 | `logs/` |
| 项目入口说明 | `CLAUDE.md` |
| 业务事实、字段含义 | `quick-ref/` |
| AI 容易踩坑的约束 | `rules/` |
| 高频问题处理流程 | `ai-lab skills/` |
| 高频开发动作 | `.claude/skills/` |
| 明确违规且可静态判断 | `.claude/hooks/` |
| 复杂审查、第二视角 | `.claude/agents/` |
| 写作和维护规范 | `core/standards/` |

---

## 3. 升级路径

推荐逐步升级：

```text
日志记录
↓
多次复现后整理为 rules / quick-ref
↓
形成固定流程后整理为 ai-lab skill
↓
高频开发动作包装为 .claude skill
↓
明确静态违规升级为 hook
↓
复杂审查沉淀为 agent
```

不要跳过前面的验证阶段。

---

## 4. 什么时候放 rules

适合 rules：

- AI 可能不知道。
- 项目特有。
- 容易踩坑。
- 不是简单代码搜索就能理解。

不适合 rules：

- 通用编程常识。
- 基础 CRUD 写法。
- 已经由代码结构清楚表达的事实。
- 临时任务要求。

---

## 5. 什么时候放 quick-ref

适合 quick-ref：

- 文件位置。
- 实体关系。
- 数据流向。
- 页面/API 映射。
- 外部系统字段来源。

quick-ref 的目标是让 AI 快速定位，不是解释所有背景。

---

## 6. 什么时候做成 skill

适合 ai-lab skill：

- 高频。
- 有固定检查步骤。
- 跨任务复用。
- 容易遗漏。

适合 `.claude/skills`：

- 高频开发动作。
- 希望 Claude Code 直接识别和调用。
- 有明确触发关键词。

---

## 7. 什么时候做成 hook

适合 hook：

```text
高频 + 明确 + 可静态判断 + 继续执行风险高
```

不适合 hook：

- 需要理解完整业务链路。
- 误判会频繁打断开发。
- 规则还不稳定。
- 只是个人偏好。

---

## 8. 什么时候做成 agent

适合 agent：

- 需要第二视角。
- 需要跨文件审查。
- 输出需要结构化 findings。
- 主会话不适合承载大量上下文。

不适合 agent：

- 简单搜索。
- 单文件小改。
- 可以由 hook 明确阻断的问题。

---

## 9. 工程化配置的选择性生效

工程化配置不是写进 ai-lab 就会生效。

```text
ai-lab 中的 .claude 副本 = 学习资料 / 配置样本
真实项目中的 .claude = Claude Code 实际读取的配置
```

因此可以放心在 ai-lab 中总结、比较、裁剪配置经验；只有当你把某个配置复制到目标项目对应目录，并调整为真实 `settings.json`、`hooks/`、`skills/` 后，它才会参与运行。

---

## 10. 维护原则

```text
先记录，再归纳，再模板化，最后才自动化。
```

每次新增规则都问：

- 这是不是项目特有？
- 是否已经多次复现？
- 是否能从代码直接看出来？
- 是否需要运行时强制？
- 放到哪里未来最容易找到？
