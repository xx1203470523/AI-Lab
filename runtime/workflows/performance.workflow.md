# Workflow: 性能诊断

## 模型分工

性能问题可以两条线并行，最后汇总结论：

```
Flash (analyst)                   Pro (analyst)
    │                                  │
    │  Step 1a: 快速诊断               │  Step 1b: 深度分析
    │  - 收集慢日志                    │  - SQL 执行计划
    │  - 定位可疑接口                  │  - 索引深度分析
    │  - 检查 N+1 查询                 │  - 锁竞争分析
    │  - 检查内存泄漏                  │  - OOM 根因
    │                                  │
    ├────────── 汇总 ─────────────────┤
    │                                  │
    │          Step 2: 方案评审（Pro 主导）
    │          - 对比两方诊断
    │          - 确定根因
    │          - 给出优化方案
    │                                  │
    │  Step 3: 方案落地（Flash 执行）   │
    │  - 加索引                        │
    │  - 改 SQL                        │
    │  - 优化代码                      │
    │                                  │
    ├────── 验证请求 ─────────────────→│
    │                                  │  Step 4: 效果验证（Pro）
    │  ←────── 验证结果 ───────────────┤  - 对比优化前后
    │                                  │  - 确认收益
```

## 职责交叉说明

同一个慢接口，Flash 和 Pro 在不同深度同时分析：

| 分析维度 | Flash（Step 1a） | Pro（Step 1b） | 交叉互补 |
|----------|-----------------|---------------|----------|
| SQL 查询 | 提取 SQL，检查 SELECT *、缺失 WHERE | 执行计划分析、索引建议、锁粒度 | Flash 定位问题 SQL，Pro 深度优化 |
| 内存 | 检查是否全量加载到 List、DataTable | 分析 GC 压力、大对象堆碎片 | Flash 找代码问题，Pro 找运行时问题 |
| 连接池 | 检查是否未释放连接 | 分析连接池耗尽根因 | 同上 |
| 缓存 | 检查是否有热点数据重复查库 | 设计缓存策略（TTL、淘汰） | Flash 找缺失，Pro 定方案 |
| 分页 | 检查是否缺分页 | 评估大数据量流式方案 | Flash 看有没有，Pro 看好不好 |

> 关键：Step 1a 和 Step 1b **无依赖，可并行执行**——两个窗口同时开工。

## 诊断清单

### Step 1a — Flash 快速诊断

#### 代码层面
- [ ] 接口入口是否记录耗时
- [ ] 定位最慢的数据库调用
- [ ] 是否存在 N+1 查询（循环内查库）
- [ ] 是否存在 SELECT *（全列查询）
- [ ] 是否缺少分页
- [ ] 是否全量加载到内存后再处理
- [ ] 导出是否使用流式写入（MiniExcel）

#### 收集信息
- [ ] 慢日志截图/片段
- [ ] 数据量估算（表行数）
- [ ] 现有索引列表
- [ ] 关联 Service/Repository 文件路径

### Step 1b — Pro 深度分析

#### SQL 深度
- [ ] 获取执行计划（EXPLAIN / SET STATISTICS）
- [ ] 分析索引命中情况（全表扫描 / Index Seek）
- [ ] 分析 JOIN 顺序和方式（Nested Loop / Hash Join）
- [ ] 评估锁竞争（Shared Lock / Exclusive Lock）
- [ ] 子查询改写为 JOIN 的可能性

#### 架构层面
- [ ] 是否需要读写分离
- [ ] 是否需要汇总表/物化视图
- [ ] 是否需要异步/队列化解耦
- [ ] 是否需要引入二级缓存

### Step 2 — 方案评审（Pro 主导）
- [ ] 对比 Flash 和 Pro 的诊断，确认根因
- [ ] 给出优化方案（标注预期收益和风险）
- [ ] 给出索引变更 DDL（标注对写入的影响）

### Step 3 — 方案落地（Flash 执行）
- [ ] 按 Pro 方案修改代码
- [ ] 添加索引（先在测试环境验证）
- [ ] 编译通过

### Step 4 — 效果验证（Pro）
- [ ] 对比优化前后响应时间
- [ ] 确认无性能退化
- [ ] 确认无新问题引入

```markdown
### 综合方案（Step 2）

|------|--------|--------|------|
```

## 使用方式

```
# Flash 窗口
@workflow 创建 performance
问题：报表"月度入库汇总"导出 OOM
数据量：约 50w 行

# Pro 窗口（可同时进行）
@workflow 处理 active/xxx-performance.md step 1b

# 汇总后 Flash 窗口
@workflow 处理 active/xxx-performance.md step 3
```
