# Workflow: 代码审查

## 模型分工

同一份变更，三个角色从不同深度参与：

```
Flash (author)          Pro (reviewer)          Flash (executor)
    │                       │                       │
    │  Step 1: 快速自查      │                       │
    │  - 语法检查            │                       │
    │  - 明显逻辑错误        │                       │
    │  - 规范自检            │                       │
    │                       │                       │
    ├── 提交审查 ───────────→│                       │
    │                       │  Step 2: 深度审查      │
    │                       │  - 并发安全            │
    │                       │  - 事务边界            │
    │                       │  - 数据一致性          │
    │                       │  - 安全漏洞            │
    │                       │  - 对比 Step1 找遗漏   │
    │                       │                       │
    │  Step 3: 修改代码 ←───┤  写回审查报告          │
    │  - 修复严重问题        │                       │
    │  - 处理建议改进        │                       │
    │                       │                       │
    ├── 请求重审 ───────────→│                       │
    │                       │  Step 4: 重审修改      │
    │                       │  - 确认修复            │
    │                       │  - 检查新问题          │
    │                       │                       │
    │  确认通过 ←───────────┤  Step 5: 最终结论      │
```

## 职责交叉说明

审查同一段代码时，Flash 和 Pro 视角互补：

| 检查项 | Flash（Step 1） | Pro（Step 2） | 为什么交叉 |
|--------|----------------|---------------|------------|
| 并发安全 | 检查 lock/using 是否缺失 | 分析竞态条件、死锁可能性 | Flash 看语法，Pro 看逻辑 |
| 事务边界 | 检查 TransactionScope 是否存在 | 分析事务范围合理性、跨库补偿 | Flash 看有没有，Pro 看好不好 |
| SQL 查询 | 检查是否拼接 SQL | 分析执行计划、锁粒度 | Flash 看安全，Pro 看性能 |
| 异常处理 | 检查是否吞异常 | 分析异常恢复策略 | Flash 看代码，Pro 看设计 |
| 库存操作 | 检查扣减调用是否正确 | 分析并发扣减一致性 | Flash 看调用链，Pro 看时序 |

## 审查清单

### Step 1 — Flash 快速自查（author 视角）
- [ ] 编译是否通过
- [ ] 命名是否符合项目规范
- [ ] 是否有明显的 null 引用风险
- [ ] 是否有未使用的变量/引入
- [ ] 日志是否完整（入口+关键分支+异常）
- [ ] 权限注解是否正确
- [ ] API 响应是否走 `Success()` / `ToResponse()`

### Step 2 — Pro 深度审查（reviewer 视角）

#### 并发安全
- [ ] 共享状态是否有锁保护
- [ ] 库存扣减是否存在竞态条件
- [ ] 分布式锁使用是否正确（key 粒度、超时时间）
- [ ] async/await 是否正确传播

#### 事务边界
- [ ] 事务范围是否合理（不过大不过小）
- [ ] 跨库操作是否有事务补偿
- [ ] 事务内是否有外部 HTTP/RPC 调用

#### 数据一致性
- [ ] 写操作后是否刷新缓存
- [ ] 多表更新是否在同一事务内
- [ ] 软删除过滤条件是否正确

#### 安全
- [ ] 是否有 SQL 注入风险
- [ ] 权限检查是否完备
- [ ] 敏感数据是否脱敏

#### 对比 Step 1
- [ ] Flash 自查是否有遗漏（重点标注）

### Step 3 — Flash 修改执行
- [ ] 所有严重问题已修复
- [ ] 建议改进逐条处理（修复或说明不修原因）
- [ ] 修改后编译通过

### Step 4 — Pro 重审
- [ ] 逐条确认修复
- [ ] 检查修改是否引入新问题
- [ ] 不通过 → 回到 Step 3，记录迭代轮次

### Step 5 — 最终结论
- 风险等级：高/中/低
- 合并决策：通过 / 条件通过 / 不通过

## 使用方式

### Flash 侧（发起审查）

```
@workflow 创建 code-review

审查范围：当前分支 feature/xxx 相对于 main 的变更
重点关注：并发安全、事务边界
关联规则：concurrency.md, wms-business.md
```

### Pro 侧（执行深度审查）

```
@workflow 处理 active/20260511-xxx-code-review.md step 2
```

### 迭代重审

```
@workflow 处理 active/20260511-xxx-code-review.md step 4
```
