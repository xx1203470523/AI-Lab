# Workflow: 架构设计

## 模型分工

架构设计是典型的「Flash 发散 → Pro 收敛」协作模式：

```
Flash (drafter)                   Pro (reviewer)
    │                                  │
    │  Step 1: 需求梳理                │
    │  - 收集需求背景                  │
    │  - 梳理现有模块关系              │
    │  - 列出约束条件                  │
    │                                  │
    ├── 需求文档 ─────────────────────→│
    │                                  │  Step 2: 方案设计
    │                                  │  - 领域建模
    │                                  │  - 分层设计
    │                                  │  - 数据模型
    │                                  │  - 风险评估
    │                                  │
    │  Step 3: 方案质疑 ←─────────────┤
    │  - 挑战 Pro 的假设              │
    │  - 提出替代方案                  │
    │  - 补充业务边界情况              │
    │                                  │
    ├── 质疑意见 ─────────────────────→│
    │                                  │  Step 4: 方案定稿
    │                                  │  - 回应质疑
    │                                  │  - 调整方案
    │                                  │  - 输出最终方案
    │                                  │
    │  Step 5: 实施执行                │
    │  - 按方案分步落地                │
    │  - 遇到问题记录                  │
    │                                  │
    ├── 实施反馈 ─────────────────────→│
    │                                  │  Step 6: 方案调整
    │                                  │  - 根据实施反馈微调
```

## 职责交叉说明

同一个架构决策，Flash 和 Pro 从不同角度参与：

| 环节 | Flash（drafter） | Pro（reviewer） | 交叉点 |
|------|-----------------|-----------------|--------|
| 领域建模 | 梳理现有实体关系 | 设计新聚合根、值对象 | Flash 提供现状，Pro 设计未来 |
| 接口设计 | 列出需要的 API 端点 | 评审 RESTful 语义、兼容性 | Flash 提需求，Pro 审设计 |
| 数据模型 | 整理现有表结构 | 设计新表+索引+迁移 | Flash 提供约束，Pro 做设计 |
| 风险评估 | 补充业务边界情况 | 识别技术风险 | Flash 懂业务，Pro 懂技术 |
| 方案质疑 | 挑战 Pro 的假设 | 回应质疑、调整方案 | 双向互动，共同打磨 |

## 设计步骤

### Step 1 — 需求梳理（Flash）

- [ ] 明确业务目标（要解决什么问题）
- [ ] 梳理现有模块关系和调用链
- [ ] 列出技术约束（框架、数据库、现有接口兼容）
- [ ] 列出业务约束（现有流程不能中断、PDA 兼容等）
- [ ] 标注不确定项（需要 Pro 帮助决策的点）

### Step 2 — 方案设计（Pro）

#### 领域建模
- [ ] 新增实体/值对象/聚合根
- [ ] 与现有实体的关系
- [ ] 枚举扩展

#### 分层设计
- [ ] Controller → Service → Repository 调用链
- [ ] 是否复用现有 Service
- [ ] 跨模块依赖是否合理

#### 数据模型
- [ ] 新表 DDL
- [ ] 索引设计
- [ ] 迁移策略（是否需要预迁移）

#### 风险评估
- [ ] 并发冲突
- [ ] 数据一致性
- [ ] 性能影响
- [ ] 回滚难度

### Step 3 — 方案质疑（Flash 反审 Pro）

Flash 从**业务熟悉度**角度挑战 Pro 的方案：

- [ ] 这个设计是否和现有业务逻辑冲突？
- [ ] 是否有 Pro 不知道的业务边界情况？
- [ ] 是否有更简单的替代方案？
- [ ] 实施复杂度是否可接受？

> 这一步很重要：Pro 强在推理深度，但不如 Flash 熟悉项目细节。Flash 的反审能补上业务视角的盲区。

### Step 4 — 方案定稿（Pro）

- [ ] 回应 Flash 的质疑
- [ ] 调整方案（如需要）
- [ ] 输出最终设计文档

### Step 5 — 实施执行（Flash）

- [ ] 按方案分步实施
- [ ] 每步完成后自查
- [ ] 遇到未预见问题及时反馈

### Step 6 — 方案调整（Pro）

- [ ] 根据实施反馈微调方案
- [ ] 记录决策变更及原因

## 输出格式

```markdown
## 架构设计方案

### 需求理解（Step 1 — Flash）
- 业务目标：
- 现有架构约束：
- 不确定项（需 Pro 决策）：

### 设计方案（Step 2 — Pro）
- 领域建模：
- 分层设计：
- 数据模型：

### 质疑与回应（Step 3 & 4）
| Flash 质疑 | Pro 回应 | 是否调整方案 |
|------------|----------|--------------|

### 风险矩阵
| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|

### 实施步骤（Step 5）
| 步骤 | 内容 | 涉及文件 | 状态 |
|------|------|----------|------|

### 实施反馈与调整（Step 6）
| 反馈 | 调整 | 原因 |
|------|------|------|
```

## 使用方式

```
# Flash 窗口
@workflow 创建 architecture
需求：新增"质检批次管理"，在入库流程中拦截待检物料
约束：不影响现有 ASN→收货→上架流程

# Pro 窗口
@workflow 处理 active/xxx-architecture.md step 2

# Flash 反审（回到 Flash 窗口）
@workflow 处理 active/xxx-architecture.md step 3

# Pro 定稿
@workflow 处理 active/xxx-architecture.md step 4
```
